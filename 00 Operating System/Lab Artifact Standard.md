# Lab Artifact Standard

## Purpose

Every lab must leave behind artifacts that make the learning reusable.

An artifact is useful when it can answer:

- What claim did this lab test?
- What evidence did it use?
- What output did it produce?
- How was the output judged?
- What failed?
- What can be reused elsewhere?

## Required Artifacts Per Lab

Each active lab should produce:

1. Lab brief
2. Loop spec
3. Input examples
4. Expected outputs or scoring rubric
5. Actual outputs
6. Scorecard or eval notes
7. Failure-modes note
8. Reusable playbook
9. Teach-back artifact
10. Weekly review entry

## Minimum Viable Lab

A lab is minimally complete when it has:

- one clear claim type
- at least five examples
- an expected result for each example
- one generated or manually produced output
- one failure-mode note
- one decision about what to do next

## Preferred 30-Day Lab Shape

For a full 30-day sprint, aim for:

- one loop spec
- one tiny local working version or manual workflow
- one eval set with about 20 examples
- one failure-modes note
- one markdown report output
- one 5-minute teach-back script
- one weekly review record

## Artifact Quality Bar

An artifact is good enough when:

- it is understandable without a live explanation
- it uses sanitized or synthetic examples
- it separates claims from evidence
- it names uncertainty
- it identifies next evidence needed
- it can be reused or adapted in another project

## Artifact Anti-Patterns

Avoid:

- notes that summarize activity but do not preserve evidence
- reports that make stronger claims than the source supports
- mock data that is secretly real data with names removed
- evals without expected outputs
- playbooks that require undocumented context
- automation that hides human approval boundaries
